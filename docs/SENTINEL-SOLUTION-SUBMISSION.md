# Submitting to Azure-Sentinel Content Hub

This document describes how to submit XdrLogRaider as an official Microsoft Sentinel Solution to the [Azure/Azure-Sentinel](https://github.com/Azure/Azure-Sentinel) repo for publication in the Content Hub.

## Status

**Target for v1.2.** v1.0 ships as a self-hosted "Deploy to Azure" solution via raw GitHub URLs. After field-testing and community feedback, submit to Content Hub.

## Prerequisites before submitting

### Code readiness

- [ ] v1.0 has been deployed on ≥3 independent tenants successfully
- [ ] ≥3 months of operational data showing reliable ingestion
- [ ] No unresolved `portal-endpoint-broken` issues open
- [ ] All 6 workbooks field-tested by real SOC analysts
- [ ] All 15 analytic rules have been tuned against false positives in a production environment
- [ ] Security review completed (internal or contracted)
- [ ] Unit test coverage meets targets (see `CONTRIBUTING.md`)

### Legal / process readiness

- [ ] Microsoft [CLA](https://cla.microsoft.com/) signed by all contributors listed in CODEOWNERS
- [ ] Publisher identity established — Azure Sponsorship or Partner Center account
- [ ] Content Hub publisher agreement reviewed
- [ ] Privacy impact assessment if ingesting identity data (P5 Identity streams)

### Technical readiness

- [ ] `solution/` directory structure matches [Azure-Sentinel/Solutions/](https://github.com/Azure/Azure-Sentinel/tree/master/Solutions) conventions
- [ ] `manifest.json` populated per [Microsoft Sentinel Solution manifest schema](https://github.com/Azure/Azure-Sentinel/blob/master/Tools/Create-Azure-Sentinel-Solution/V3/README.md)
- [ ] Solution generator runs clean: `.github/workflows/package-solution.yml` produces a valid `<version>.zip`
- [ ] mainTemplate.json passes ARM-TTK with zero errors (not just warnings)
- [ ] createUiDefinition.json passes schema validation
- [ ] Data Connector has logo PNG/SVG (≥ 90×90 px, transparent background)
- [ ] Analytic rules in YAML format per Sentinel spec (id, name, description, severity, queryFrequency, queryPeriod, triggerOperator, triggerThreshold, tactics, relevantTechniques, query, entityMappings)
- [ ] Workbooks have YAML metadata sidecar (`<Name>.yaml` alongside `<Name>.json`) with id, title, publisher, description, dataTypesDependencies, previewImagesFileNames
- [ ] Hunting queries in YAML per [official schema](https://github.com/Azure/Azure-Sentinel/blob/master/Detections/Schemas/DetectionsSchema.json)
- [ ] Parsers as savedSearches resources in ARM

├── Analytic Rules/
│   ├── AsrRuleDowngrade.yaml
│   ├── LrUnsignedScriptsOn.yaml
│   └── ...                               # 15 rules, YAML format
├── Data Connectors/
│   └── XdrLogRaider_DataConnector.json  # CCF-style connector definition
├── Hunting Queries/
│   ├── ConfigChangesByUpn.yaml
│   └── ...                               # 10 queries, YAML format
├── Parsers/
│   └── MDE_Drift_P0Compliance.yaml       # One parser YAML per category

│   ├── MDEComplianceDashboard.json

│   ├── MDEDriftReport.json

├── Package/
│   ├── 1.0.0.zip                         # Output of solution generator

## PR submission checklist

Follow [Azure-Sentinel's CONTRIBUTING.md](https://github.com/Azure/Azure-Sentinel/blob/master/CONTRIBUTING.md) plus:

- [ ] Fork `Azure/Azure-Sentinel`, branch from `master`
- [ ] Create branch named `solutions/add-xdrlograider-<version>`
- [ ] Copy our `Sentinel-Solution/` directory into `Solutions/XdrLogRaider/` in the fork
- [ ] Update `Solutions/XdrLogRaider/Data/Solution_XdrLogRaider.json` with correct paths
- [ ] Run the official solution generator locally to regenerate `Package/`:
  ```powershell
  cd Tools/Create-Azure-Sentinel-Solution/V3
  ./createSolutionV3.ps1 -path ../Solutions/XdrLogRaider
  ```
- [ ] Verify `Package/<version>.zip` was created and is valid
- [ ] Run `Package/` validation:
  ```powershell
  ./validatePackage.ps1 -path ../Solutions/XdrLogRaider
  ```
- [ ] Deploy the generated `mainTemplate.json` to a test Sentinel workspace
- [ ] Screenshot each workbook and attach to PR
- [ ] Run `Test-ConnectorJsonSchema` on data connector JSON
- [ ] Run `Test-WorkbookJson` on each workbook
- [ ] Open PR with title `[Solution] Add XdrLogRaider v<version>`
- [ ] PR description covers: summary, included artifacts, testing evidence, dependencies, data-connector requirements
- [ ] Sign Microsoft CLA when the bot prompts
- [ ] Request review from `@Azure/sentinel-solutions-content`

## Package generator

Our repo includes `.github/workflows/package-solution.yml` which mirrors the Sentinel solution generator output. Run locally:

```powershell
./tools/Build-SentinelSolution.ps1 -Version 1.0.0 -OutputPath ./Sentinel-Solution
```

This produces the exact folder structure Azure-Sentinel expects, ready to be copied into a fork.

## Known PR gotchas

1. **Analytic rule YAML — `id` must be a GUID** — generator rejects sequential IDs. Regenerate via `[guid]::NewGuid()` per rule.
2. **Workbook `fromTemplateId`** — must match the pattern `sentinel-<WorkbookName>`; different from the Content Hub ID.
3. **Parser naming** — savedSearch `name` must match the KQL function name exactly; our convention `MDE_Drift_P<N><Category>` works.
4. **Data connector `graphQueries`** — the KQL in connector UI status queries can't reference custom tables that don't exist yet; use the heartbeat table only.
5. **CLA bot** — check PR comments if merge is blocked; sometimes requires `@msftbot rerun-validation`.
6. **ARM-TTK warnings** — warnings are acceptable, errors block merge. We aim for zero of each.
7. **Preview images** — must be ≤ 200 KB, PNG, max 1280px wide. Multiple screenshots supported.
8. **Solution manifest `version`** — semver, must exactly match Package/\<version>.zip filename.
9. **Branch-naming** — `solutions/add-<name>` or `solutions/update-<name>-<version>`; the Microsoft reviewer automation relies on this prefix.

## Post-merge

After Azure-Sentinel team approves + merges:

1. Azure pipeline publishes to Content Hub within ~3-5 business days
2. Solution becomes visible in Sentinel → Content Hub
3. Users can install with one click
4. We get analytics on install counts + rule enablement via publisher dashboard

## Maintenance after publication

- Version bumps require a new PR per change set; don't hotfix published versions in place
- Follow [Azure-Sentinel's solution update guidance](https://github.com/Azure/Azure-Sentinel/blob/master/.script/tests/Readme.md)
- Each release must regenerate `Package/<new-version>.zip`
- Deprecate broken streams rather than removing (mark in README, leave in place, add issue tracking)
